# PLAN-20260507-rbac-m1

## 1. 任务背景

已完成注册/登录/JWT 基线，但缺少角色字段与权限控制，无法支撑管理员能力与前后端联调中的权限验证。

## 2. 目标与验收结果

- 目标 1：用户模型支持角色（admin/editor/guest）。
- 目标 2：登录态接口返回角色信息。
- 目标 3：提供管理员专用接口并完成权限校验。

验收标准（必须可验证）：
- [x] `/api/v1/auth/me` 返回 `role`
- [x] 新增管理员接口（用户列表）并在非 admin 场景返回 403
- [x] 测试覆盖注册、登录、权限拒绝与权限通过路径

## 3. 本次范围 / 非范围

### 3.1 In Scope
- 数据库 users 表增加 role 字段
- 注册逻辑分配默认角色（首个用户 admin，其余 editor）
- 新增管理员接口 `GET /api/v1/auth/users`

### 3.2 Out of Scope
- 不实现复杂 RBAC 权限矩阵
- 不实现角色变更后台管理接口

## 4. 技术方案

- 扩展数据库初始化与仓储层查询字段
- 在 auth endpoint 增加角色校验函数
- 使用现有 JWT 机制承载登录态，不额外引入中间件

## 5. 任务拆解与推进顺序

1. 数据层扩展：users role 字段 + 仓储方法
2. 接口层扩展：register/login/me/users
3. 测试与文档更新

## 6. 风险与回滚

- 风险点：旧库缺 role 列导致查询失败
- 监控点：启动建表/迁移是否成功
- 回滚方案：仅新增字段与接口，可按文件回退

## 7. 验证方案

- 单元/接口测试：pytest 全量通过
- 手工验证路径：register -> login -> me -> users(admin/non-admin)
- 边界场景：缺 token、无权限、重复注册

## 8. 交付清单

- 代码文件：auth endpoint、repository、database、tests
- 文档文件：README（认证说明补充）
- 变更说明：完成 M1 的最小可用角色权限能力
