# 系统升级与高可用落地详细设计方案 (30万级别图片规模)

## 整体实施路径
分为三个阶段进行演进：
1. **阶段一**：存储与任务解耦（奠定大数据量基础）
2. **阶段二**：核心组件高可用升级（消除单点故障）
3. **阶段三**：性能优化与成本控制（应对极高并发访问）

---
## 阶段一：存储与任务解耦

### 步骤 1.1：关系型数据库迁移
**实现内容**：将元数据库从单机 SQLite 迁移至支持高并发读写的分布式/主从关系型数据库，并使用 SQLAlchemy/Alembic 完成数据平滑迁移。
**方案选择**：
*   **方案 A：云厂商托管 MySQL (RDS)**
    *   *利*：开箱即用，自动备份，主从一键切换，运维成本极低。
    *   *弊*：需要长期支付云服务费用。
    *   *推荐度*：⭐⭐⭐⭐⭐
*   **方案 B：自建 PostgreSQL/MySQL 集群**
    *   *利*：成本可控，数据自主权最高。
    *   *弊*：需要专业的 DBA 运维，高可用机制（如 Keepalived+HAProxy 或 Patroni）搭建复杂。
    *   *推荐度*：⭐⭐⭐
**推荐方案**：方案 A (云托管 MySQL/RDS)
**推荐方案潜在不足**：受限于云厂商规格，未来若数据量涨至千万级可能遇到单表性能瓶颈；云服务 Vendor Lock-in (供应商锁定)。
**应对策略**：在代码层坚持使用标准 ORM（SQLAlchemy），避免使用厂商特有 SQL 语法；表设计时预留水平分表字段（如根据租户或月份分表）。
**验收结果**：代码无缝切换至 MySQL，原有 SQLite 数据 100% 无损导入；支持至少 2000 QPS 的并发元数据读写而不报错。

### 步骤 1.2：图片文件存储迁移至对象存储 (OSS)
**实现内容**：废弃本地磁盘+Nginx读图模式。图片统一上传至阿里云 OSS / AWS S3，数据库仅保存图片对象 Key 或 URL。
**方案选择**：
*   **方案 A：后端代传 (Client -> FastAPI -> OSS)**
    *   *利*：代码改动极小，安全性高，完全在后端控权。
    *   *弊*：后端带宽和 CPU 压力翻倍，上传大文件容易超时，不适合 30万+ 图片高并发上传。
    *   *推荐度*：⭐⭐
*   **方案 B：前端直传 OSS (Client -> OSS，后端仅签发 Token)**
    *   *利*：解放后端带宽，上传速度最快（直接利用云厂商边缘节点），支持海量并发上传。
    *   *弊*：前端代码改动大，需引入 STS 临时凭证机制，复杂度上升。
    *   *推荐度*：⭐⭐⭐⭐⭐
**推荐方案**：方案 B (前端直传 OSS + STS 凭证)
**推荐方案潜在不足**：存在恶意用户刷流、刷上传接口消耗 OSS 容量和流量资费的风险。
**应对策略**：后端在签发 STS Token 时，必须严格限制每个用户/IP的上传频率（Rate Limit）；限制 STS Token 的权限边界（只能上传到指定前缀目录）和过期时间（如 5 分钟）；OSS 设置跨域规则(CORS)和防盗链。
**验收结果**：关闭服务器本地磁盘写入权限后，系统仍能正常完成图片上传并在前端展示；后端网卡上行/下行流量下降 90%。

### 步骤 1.3：引入异步任务队列解耦 AI 处理
**实现内容**：图片上传后，立刻在 DB 生成 `pending` 状态的记录，通过消息队列通知 Worker 后台异步执行大模型 (Qwen3-VL) 分析和 Embedding 生成。
**方案选择**：
*   **方案 A：Celery + Redis**
    *   *利*：Python 生态最成熟，易于集成到 FastAPI，Redis 既可做 Broker 也可做 Result Backend。
    *   *弊*：如果任务量极端巨大（百万级积压），Redis 作为 Broker 可能会导致内存爆满或丢失任务。
    *   *推荐度*：⭐⭐⭐⭐
*   **方案 B：Celery + RabbitMQ (或 Kafka)**
    *   *利*：专业的消息代理，支持消息持久化，确保任务绝对不丢失，适合海量任务堆积。
    *   *弊*：需额外额外部署和维护 RabbitMQ/Kafka 组件。
    *   *推荐度*：⭐⭐⭐⭐⭐
**推荐方案**：方案 B (Celery + RabbitMQ)
**推荐方案潜在不足**：Worker 消费速度如果低于上传速度，任务会无限积压；调用大模型 API 频率过高会导致被云服务商限流 (Rate Limit)。
**应对策略**：
1. Celery 层面实现严格的限流机制（Rate Limiting），例如限制调用大模型的 Task `rate_limit='10/s'`。
2. 配置死信队列与失败重试策略（指数退避重试）。
3. 监控 RabbitMQ 队列深度，结合 K8s HPA 自动弹缩 Worker 节点数量。
**验收结果**：100 用户并发同时上传 1000 张图片时，接口 0.5 秒内响应成功；后台队列稳定积压并按照设定速率消费，无任何一笔 API 调用因“并发过高”报错失败。

## 阶段二：核心组件高可用升级

### 步骤 2.1：向量数据库高可用集群化
**实现内容**：将单机版 Qdrant 升级为分布式 Qdrant Cluster（或 Milvus），支持数据分片和多副本，支撑 30万+ 向量（约 1024 维）的高并发检索。
**方案选择**：
*   **方案 A：Qdrant Cloud 托管服务**
    *   *利*：免除所有部署、运维、扩容烦恼，SLA 有保障。
    *   *弊*：持续的云端使用成本（尤其对内存和 IO 要求高的向量库，托管费用不菲）。
    *   *推荐度*：⭐⭐⭐⭐
*   **方案 B：在 Kubernetes (K8s) 上自建 Qdrant Cluster**
    *   *利*：硬件成本最低（复用已有集群服务器资源）。
    *   *弊*：向量数据库的 StatefulSet 部署、持久化存储(PV/PVC)分配、节点容灾恢复的运维门槛极高。
    *   *推荐度*：⭐⭐⭐
**推荐方案**：方案 A (Qdrant Cloud 托管服务) —— 前提是预算允许；若预算受限则选方案 B。
**推荐方案潜在不足**：云服务间的网络延迟；若业务量激增，Qdrant Cloud 的阶梯计费可能导致超支。
**应对策略**：部署在与业务后端 (FastAPI) 相同的公有云可用区 (AZ) 内，使用内网互通消除延迟；定期监控索引大小和 QPS 峰值，开启阈值告警防超支。
**验收结果**：在含有 30 万条真实向量数据的库中，100 并发执行 KNN 相似度搜索，P95 延迟控制在 100 毫秒以内；随机重启其中一个节点，搜索服务不中断。

### 步骤 2.2：应用层容器化与集群部署
**实现内容**：将 FastAPI 和 Vue 彻底无状态化并打成 Docker 镜像，通过 Kubernetes (K8s) 部署多副本，由 Ingress/LoadBalancer 统一引流。
**方案选择**：
*   **方案 A：Docker Swarm / Docker Compose (多台虚拟机互联)**
    *   *利*：学习曲线平缓，适合小团队起步。
    *   *弊*：生态老旧，难以实现精细化的自动扩缩容、滚动更新和可观测性。
    *   *推荐度*：⭐⭐
*   **方案 B：Kubernetes (K8s) + Helm**
    *   *利*：业界标准，支持基于 CPU/内存甚至自定义指标的自动扩容 (HPA)，支持平滑蓝绿/金丝雀发布。
    *   *弊*：K8s 集群维护成本高，需专业的 DevOps 技能。
    *   *推荐度*：⭐⭐⭐⭐⭐
**推荐方案**：方案 B (依托云厂商的托管 K8s，如 ACK / EKS)
**推荐方案潜在不足**：如果应用尚未完全“无状态化”（如依然向本地写日志或缓存），会导致 K8s Pod 漂移时数据丢失。
**应对策略**：严格执行 12-Factor App 规范：日志全部输出到 stdout 并通过 Fluentd/Filebeat 统一收集到 ELK 平台；所有本地状态全部剔除，缓存迁移至 Redis。
**验收结果**：通过压测工具 (如 JMeter) 压测 API 网关，观测到 K8s 能够根据 CPU 负载自动从 2 个 Pod 扩容至 10 个 Pod；Kill 掉任意一个 Pod，业务请求成功率维持在 99.99% 以上。

## 阶段三：性能优化与成本控制

### 步骤 3.1：热点数据缓存引入 (Redis)
**实现内容**：将 Agent 搜索对话的中间状态、高频搜索词结果、API 鉴权 Token 等引入 Redis 集群缓存。
**方案选择**：
*   **方案 A：应用内部缓存 (如 LRUCache / Memory)**
    *   *弊*：各 Pod 数据不一致，内存碎片化，不适合集群环境。
*   **方案 B：云托管 Redis Cluster**
    *   *利*：高可用、主从切换、持久化、数据分片完善。
    *   *推荐度*：⭐⭐⭐⭐⭐
**推荐方案**：云托管 Redis Cluster。
**推荐方案潜在不足**：存在缓存击穿（热点 Key 失效瞬间 DB 被打挂）、缓存穿透（恶意查不存在的词）的风险。
**应对策略**：热点数据设置永不过期（后台异步刷新）；对于空查询结果也设置短暂的空缓存；为不同数据设置随机的过期时间，防止雪崩。
**验收结果**：重复查询相同的文本检索指令，系统不会重复调用大模型 Agent API，响应时间从数秒下降到 20 毫秒以内。

## 综合落地成本分析

为了在生产环境中实际落地这套支撑 30万+ 图片的高可用架构，需要综合考量以下三方面的成本：

### 1. 资金成本 (Financial Cost) - 按月估算 (以国内主流公有云为例)
*   **计算资源 (K8s 集群 Nodes)**：约 3-4 台 8核16G 云服务器，约 1000 - 1500 元/月。
*   **数据库 (云 RDS MySQL 架构版)**：约 4核8G，约 500 - 800 元/月。
*   **对象存储 (OSS + CDN)**：500GB ~ 1TB 存储约 100 元/月，外网流出/CDN下行流量费约 300 - 500 元/月（高度取决于用户访问频次）。
*   **大模型调用费 (Bailian Qwen3-VL/qwen-plus)**：30万图片全量处理一遍，若单张图片按 0.01 元计（需根据官方实际定价核算），初始化一次性需约 3000 元；后续增量计费及日常 Agent 对话费视活跃度而定。
*   **消息中间件/缓存/向量库 (云端托管实例)**：Redis (约100元) + RabbitMQ (约200元) + Qdrant Cloud (预估400元) = 约 700 元/月。
*   **汇总预估**：日常固定运转每月约 3000 - 4000 元人民币。初始化的大规模存量图片 AI 处理成本约数千元。

### 2. 研发成本 (Engineering Cost)
*   **后端开发 (1-2名 Python 工程师)**：需进行全方位的异步改造（接入 Celery）、ORM 迁移、对接 OSS SDK、改写 Agent 限流与重试机制。预计工期：**3-4 周**。
*   **前端开发 (1名 Vue 工程师)**：需接入 OSS STS 直传机制、调整文件预览地址。预计工期：**1 周**。
*   **运维/架构开发 (1名 DevOps)**：K8s 集群搭建、CI/CD 部署流水线配置、监控告警 (Prometheus + Grafana) 搭建。预计工期：**2 周**。
*   **总工时评估**：约 1 - 1.5 个月可完成测试环境落地、数据迁移脚本编写及压测验收。

### 3. 运维成本 (Operation Cost)
*   **复杂度升级**：由原本单一的“重启 Python 脚本”转变为对整个 K8s 微服务生态、消息队列积压状况、数据库主从同步的全面监控。
*   **应急场景**：需具备处理大模型 API 服务商突发宕机的兜底策略（如队列暂停、告警通知）、数据的自动备份与恢复演练、以及应对各类告警风暴的能力。
*   **配套工具要求**：强烈推荐引入自动化巡检体系和全链路追踪 APM（应用性能监控，如 SkyWalking/ARMS），快速定位高并发下的性能瓶颈。

> [!TIP]
> **落地建议：**
> 这是一次从“单机系统”到“企业级生产系统”的质变。建议在实际开发时，**优先实现第一阶段的“OSS 图片直传与 Celery 异步队列解耦”**。这一步的研发投入最低，但解决的性能痛点（单机 IO 卡顿、大模型响应慢导致超时）最明显。等系统在异步架构下平稳运行后，再分步骤引入 K8s 容器集群和托管数据库。
