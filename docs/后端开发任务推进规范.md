# 后端开发任务推进规范（执行版）

## 1. 目的

统一后端开发节奏，确保每次开发都有清晰范围、可追踪过程、可验收结果。  
**从现在开始：每次编码前必须先提交完整 Plan 文档，确认后再实现。**

## 2. 当前后端基线

- 技术栈：FastAPI + SQLite + JWT（MVP）
- 已有能力：`/api/v1` 路由基线、注册/登录/`/auth/me`、中间件与基础测试
- 当前阶段目标：先完成素材管理主链路，再接入检索增强

## 3. 任务划分（按里程碑）

1. **M1 鉴权与权限完善**
   - JWT 刷新策略、角色权限拦截、用户管理接口
2. **M2 素材主链路**
   - 素材上传、详情、编辑、删除、分页查询
3. **M3 标签与分组**
   - 标签管理、素材打标、集合/分组管理
4. **M4 版本与分享**
   - 素材版本记录、分享链接、下载审计
5. **M5 检索增强**
   - 文本搜图、图搜图、分页与过滤
6. **M6 稳定性与运维**
   - 性能优化、日志完善、发布与回滚

## 4. 推进方式（每个任务统一流程）

1. 需求澄清：明确目标、范围、验收标准  
2. 提交 Plan：编写本次任务计划文档（必须）  
3. 计划确认：计划通过后开始编码  
4. 开发实现：按 Plan 逐项落地  
5. 自测验证：接口/测试/回归  
6. 验收交付：提交结果与偏差说明

## 5. 强制要求：每次编码必须先有 Plan

- **编码前置条件**：存在本次任务 Plan 文档，且已确认。
- **存放目录**：`docs/plans/`
- **命名规范**：`PLAN-YYYYMMDD-<topic>.md`
  - 示例：`PLAN-20260507-assets-crud.md`
- **Plan 最低内容**：
  - 任务背景与目标
  - 本次范围 / 非范围
  - 技术方案与影响面
  - 分步骤任务清单
  - 验收标准（可执行）
  - 风险与回滚方案

## 6. 验收规则（Definition of Done）

以下条件全部满足才可验收：

- 功能与 Plan 目标一致
- 接口契约清晰（请求/响应/错误码）
- 相关测试通过（至少覆盖新增关键路径）
- 文档同步更新（README / API / 注意事项）
- 提供“完成项 + 未完成项 + 风险”说明

## 7. 执行建议

- 任务粒度控制在“可在一次迭代完成并验收”的范围内
- 不跨里程碑混做，避免不可控变更
- 先保证主流程可用，再做优化项
