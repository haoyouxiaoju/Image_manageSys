# 基于 Chinese-CLIP 的企业版素材库系统：开发起步文档

## 1. 产品定位与目标边界

这首先是一个**素材管理产品**，不是一个“CLIP 演示项目”。  
**Chinese-CLIP 是检索增强能力（feature）**，核心价值仍然是：素材是否好管、好找、好协作、好分享。

首期（MVP）目标：
- 管理闭环：上传、分类、标签、版本、检索、分享、审计
- 协作闭环：团队可用、权限可控、链接可追踪
- 技术闭环：轻量部署、可扩展升级

暂不纳入首期：
- 审批流
- 多租户计费
- 视频检索（保留扩展位）

## 2. 已确认决策

- 模型路线：**Chinese-CLIP**
- 产品方向：**企业版素材库（内部使用起步）**
- 首发部署：**单机 Docker Compose**
- 首期规模：**10万以内素材**
- 向量引擎：**FAISS**
- 权限粒度：**角色级**
- 检索语言：**仅中文**
- 合规：**基础敏感内容审核**
- 登录方式：**本地账号密码**
- 素材版本策略：**仅版本，不含审批流**
- 检索分页：**每页 10 条**
- 文件范围：**图片优先，视频暂不纳入首期**

## 3. 轻量 MVP 技术栈（去重版）

```text
前端：Vue 3 + TypeScript
后端：FastAPI（Python）
数据库：SQLite（后续可升 PostgreSQL）
文件存储：本地磁盘（后续可升 MinIO）
向量检索：FAISS
模型：Chinese-CLIP
部署：Docker Compose
```

不在首期引入：Celery、Redis、Nginx（按瓶颈再加）。

## 4. 架构选择：FastAPI 还是 C++ 主后端？

结论：**MVP 阶段优先 FastAPI 主后端**；不建议一开始做 C++ 主框架 + Python 模型混合。

原因（核心）：
- Chinese-CLIP 工具链和样例主要在 Python 生态，联调和迭代更快
- C++ 主后端 + Python 模型服务会引入跨进程/跨语言协议、序列化、监控、部署复杂度
- 在 10 万以内规模，性能瓶颈通常不在 Web 框架本身，而在模型推理、IO、索引策略

可执行折中路线（推荐）：
1. MVP：FastAPI 单后端，上传后同步向量化（必要时用进程池）
2. 成长期：把模型推理拆为独立 Python 推理服务
3. 瓶颈期：仅把热点模块用 C++ 重写（预处理/后处理/批量管线）

## 5. 用户故事（用于约束功能设计）

### 故事 1：设计师找历史素材
设计师小明有 5000 张素材，想找“去年用过的蓝色科技感背景图”。  
系统应支持：中文语义检索 + 时间过滤 + 标签过滤 + 快速预览。

### 故事 2：市场部整理并分享活动图
小红上传活动照片，分组打标签后分享给供应商。  
系统应支持：批量上传、批量打标签、分组管理、受控分享链接、下载审计。

### 故事 3：运营回溯同一素材的变更
运营需要确认某张主视觉是否为最新版本。  
系统应支持：版本链查看、版本备注、版本间快速切换与回滚。

### 故事 4：管理员追踪高频素材与风险操作
管理员希望知道谁下载了哪些素材、哪些素材被高频检索。  
系统应支持：审计日志、检索日志、按用户/时间维度筛选。

### 故事 5：新成员快速复用团队资产
新同事不知道素材命名规则，只能靠描述找图。  
系统应支持：自然语言搜图、相关推荐、收藏与常用分组。

## 6. MVP 功能优先级（产品优先）

### P0（必须先做）
- 账号登录与角色权限（管理员/编辑/访客）
- 素材 CRUD（上传、编辑元数据、删除、详情）
- 标签体系（标签增删改查、素材打标）
- 素材分组（基础文件夹/集合）
- 素材版本（仅版本，不含审批）
- 分享能力（内部受控链接）
- 审计日志（上传、删除、下载、分享）

### P1（增强）
- 文本搜图（Chinese-CLIP）
- 图搜图（Chinese-CLIP）
- 检索过滤与分页（每页 10 条）
- 检索日志统计

## 7. 核心数据模型（草案）

- `users`：用户
- `roles` / `user_roles`：角色关系
- `assets`：素材主表
- `asset_versions`：版本表
- `tags` / `asset_tags`：标签体系
- `collections` / `collection_assets`：分组
- `share_links`：分享链接（有效期、权限范围）
- `asset_embeddings`：向量信息
- `search_logs`：检索日志
- `audit_logs`：审计日志

## 8. API 起步清单（草案）

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `GET /api/v1/auth/me`
- `GET /api/v1/auth/users`（管理员）
- `POST /api/assets/upload`
- `PUT /api/assets/{id}`
- `DELETE /api/assets/{id}`
- `GET /api/assets/{id}`
- `GET /api/assets`（分页、过滤）
- `POST /api/assets/{id}/versions`
- `GET /api/assets/{id}/versions`
- `POST /api/tags`
- `GET /api/tags`
- `POST /api/collections`
- `GET /api/collections`
- `POST /api/share-links`
- `GET /api/audit-logs`
- `POST /api/search/text`（分页，page_size=10）
- `POST /api/search/image`（分页，page_size=10）

## 9. 开发顺序（先产品可用，再 AI 增强）

1. 先交付 P0：管理与协作功能闭环  
2. 再接入 P1：Chinese-CLIP 检索增强  
3. 再做性能升级：SQLite->PostgreSQL、本地存储->MinIO、按需引入任务队列  

## 10. 当前结论

对这个项目，最优路径是：  
**Vue + FastAPI 轻量落地产品主体，CLIP 作为检索增强；C++ 用于后续热点加速，而不是首期主后端。**
