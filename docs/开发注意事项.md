# 开发注意事项（MVP 阶段）

## 1. 版本控制与分支策略

- `main`：可发布分支，只接收经过评审的合并。
- `dev`：日常集成分支。
- 功能分支：`feature/<模块>-<功能>`，例如 `feature/search-text`。
- 修复分支：`fix/<模块>-<问题>`，例如 `fix/assets-delete`。

合并建议：
- 禁止直接推送 `main`。
- 功能分支通过 PR 合并到 `dev`，里程碑再从 `dev` 合并到 `main`。

## 2. 提交规范（建议）

提交信息建议使用统一前缀：
- `feat:` 新功能
- `fix:` 缺陷修复
- `refactor:` 重构
- `docs:` 文档更新
- `chore:` 工程调整

示例：
- `feat(search): add text search pagination`
- `fix(asset): correct version rollback logic`

附加要求（从当前阶段起执行）：
- 每次 commit 必须写详细正文，至少说明：变更目的、变更内容、影响范围、验证方式。

## 3. 版本号策略

- 项目版本使用语义化版本：`MAJOR.MINOR.PATCH`（如 `0.1.0`）。
- API 从一开始加版本前缀：`/api/v1/...`，避免后期升级破坏兼容。
- 模型版本单独管理（如 `cnclip-base-v1`），写入 `asset_embeddings` 的版本字段。

## 4. 数据与模型文件管理

- 不提交运行期数据：上传文件、FAISS 索引、日志、SQLite 数据库。
- 不提交大模型权重到仓库（放在模型目录外部挂载或制品仓库）。
- 保持 `.gitignore` 与目录约定一致，避免误提交。

## 5. 数据库与结构变更

- 每次表结构变更必须有迁移脚本（即使是 SQLite 阶段）。
- 迁移脚本命名建议：`YYYYMMDD_HHMM_<变更说明>`。
- 涉及删除字段/表时，先走“兼容期”再清理，避免影响线上数据。

## 6. 开发与联调约束

- 先做 P0 管理功能，再做 P1 AI 检索能力。
- 新接口必须写清输入输出与错误码（至少在文档中维护）。
- 涉及权限的接口默认最小权限开放，按角色逐步放开。

## 7. 发布与回滚

- 每次发布打 tag（如 `v0.1.0`），并附变更说明。
- 发布包和配置分离，确保可快速回滚到上一版本。
- 对检索、上传、分享、删除四类核心流程优先做回归检查。
